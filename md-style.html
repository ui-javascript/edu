# 笔记风格

  • 用空格比逗号好
  • 用换行比句号好
  • 不用[xxx](http://)隐藏链接(除非太长) --> 网址本身也是一种信息
  • 不分h1-h6 --> 让工具自行分析 @lazy
  • 用分隔线/大波浪适时地划分逻辑内容
  • 每篇文章最好有至少一张图片 --> 有利于记忆(类似地标)
  • 尽量减少引用第三方来源的图片, 但对应的需要标出来源
  • 读书笔记, 只记录感触最深的地方 --> 可以先抄个目录迈出第一步

# 目录层级

  • 最多9个chapter
  • 最多5个topic
  • appendix, 应该是补充内容(建议仅1个)
// 层级上尽可能保持扁平, 点2下就应该找到那篇文章
[0]collection 
    ch9-chapter 
        topic5 
            _appendix1

// appendix如下: 
private --> 私人内容, 不方便公开的
cmpt, case/demo, tricks, steps, tag --> 组件/案例/示例/技巧总结/步骤/标签
record, idea/faq, todo/plan, advice --> 按照时间轴记录
notnow/old, offcut --> 不推荐的/过时内容
code/gist/snippets, worm --> 代码片段/文章选段
spec --> 标准规范
corner/minor --> 少数派

# 内容递进

  • core/basics 核心/基础
  • utils/components 工具
  • web/service 主体内容
  • devops 工程化
  • platform 集大成者
  • perf 极尽精华
  • arch 终极一跃
  • code/interview 末章回顾

# 项目类型

  • demo --> 偏语法或工具学习
  • tpl/template --> 可复用(框架的业务封装等)
  • cmpt/component --> 组件
  • diff/benchwork --> 比较/性能对比
  • code --> 源码解读
  • algo/algorithm --> 算法
  • arch/architecture --> 架构
// 子项目命名顺序
--> 底层环境(jdk8)
--> 构建工具(webpack) 

--> 基础框架(ssm/ssh/springboot2) 
--> 特色技术(beetl/hikaricp) 

--> 项目名称/作者/来源标记  

--> 应用场景(pc/h5/minapp/ie10) 
--> 业务定位(admin/mall/blog/door/portal)

--> 时间(20191908)
